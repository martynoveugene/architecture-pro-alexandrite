# Выбор и настройка мониторинга в системе

## Мотивация

Мониторинг позволяет следить за внутренним состоянием системы и
дает информацию для проактивных действий. Мониторинг предоставляет значения
бизнесовых и инфраструктурных метрик во времени.
Улучшает видимость системы, что означает более полное представление о работе системы.
Позволяет быстрее определить корень проблемы.
Позволяет выявлять узкие места при работе под нагрузкой.
Ускоряет рабочий процесс.


## Выбор подхода к мониторингу

1. Для мониторинга очереди выбираем метод RED.
2. Для всех API выбираем метод RED.


## Список метрик

### Конкретные метрики для очереди сообщений - расшифровка RED.

Rate: 
- message_stats.publish_details.rate — скорость публикации,
- message_stats.deliver_get_details.rate — скорость доставки.

Errors: 
- rejected messages - сообщения, которые были отклонены. 

Duration:
- consumers - количество активных потребителей
- end‑to‑end latency — показывает сквозную задержку от публикации до окончания обработки 
- queue time - время ожидания в очереди

### Конкретные метрики для API - расшифровка RED. 

Rate:
- Number of requests (RPS) for internet shop API
- Number of requests (RPS) for CRM API
- Number of requests (RPS) for MES API

Errors:
- Number of HTTP 400 for shop API
- Number of HTTP 400 for CRM API
- Number of HTTP 400 for MES API
- Number of HTTP 500 for shop API
- Number of HTTP 500 for CRM API
- Number of HTTP 500 for MES API

Duration:
- Response time (latency) for shop API
- Response time (latency) for CRM API
- Response time (latency) for MES API


## План действий

1. Поскольку RabbitMQ уже содержит метрики и нам нужны его стандартные метрики, то начнем с него.
- Настройка плагина rabbitmqreceiver для OTel Collector, который будет забирать метрики с RabbitMQ по http api
- Настройка OTel Collector для отправки метрик в Prometheus
- Настройка Grafana для отображения метрики из Prometheus

2. Реализация мониторинга в АПИ
- Подключение в каждый API библиотеку OpenTelemetry SDK
- Доработка API для отправки через SDK метрик по протоколу OTLP в OTel Collector
уже в целом сделано в рамках предыдущей задачи:
- Настройка OTel Collector для отправки метрик в Prometheus
- Настройка Grafana для отображения метрики из Prometheus
