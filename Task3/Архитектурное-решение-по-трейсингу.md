# Архитектурное решение по трейсингу

## Системы, которые следует покрыть трейсингом

Все системы в которых может меняться статус заказа, а именно:
 - онлайн-магазин
 - MES
 - CRM

RabbitMQ будет вовлечен в процесс трассировки путем передачи контекста - за счет сохранения в сообщениях заголовков трейсинга.

## Мотивация

### Общие сведения

Трейсинг — метод отслеживания запросов в приложении.
Полезен в микросервисной архитектуре, когда есть множество сервисов внутри и один сервис может вызывать другой и так по цепочке.
Трейсинг начинается с точки входа запроса в систему: он запускается для запроса и имеет уникальный идентификатор, сгенерированный для этого запроса.
Далее формируется список вызовов и этот список представляется в виде дерева.
На каждой вершине видно с какими параметрами прошёл вызов и сколько времени он занял.

Помогает в ситуации, когда нужно понять, кто, когда и в какой последовательности вызывал сервис.
Помогает понять исходный запрос для неудачного вызова, поскольку исключение или ошибка обычно фиксируется вместе со всем контекстом запроса.

### Выгода для компании

В компании внедрение трейсинга будет эффективно, поскольку поможет отследить, где теряется заказ в цепочке работы над ним.
Обеспечит обработку жалоб от клиентов на просроченные заказы.
Обеспечит обработку жалоб со стороны представители компаний, которые ежедневно сообщают, что не получили свои заказы.
В каждом случае будет выявлена конкретная корневая причина срыва работы и можно будет заводить конкретные задачи на 
конкретных людей по устранению первопричин.
Вся эта работа приведет к решению проблем с потерей контрактов и устранит недовольство клиентов.

### Метрики, на которые повлияет внедрение трейсинга

 - Количество потерянных заказов уменьшится
 - Количество ошибок не снабженных контекстом и логами уменьшится
 - Уменьшится количество багов, которые невозможно воспроизвести
 - Количество жалоб клиентов и представителей компаний уменьшится


## Предлагаемое решение

Решение будет базироваться на платформе наблюдения OpenTelemetry (OTel), которая предоставляет стандартизированный способ сбора, 
обработки и экспорта телеметрических данных-сигналов (метрик, трассировок и журналов) из распределённых систем. 
Во все API будет встроено sdk, которое будет отправлять OTLP запросы в OTel Collector.
Далее OTel Collector будет отправлять отдельно метрики в Prometheus и сами трейсы в Jaeger.
Grafana занимается отображением графиков с метриками и трейсов.

## Компромиссы

 - Для системы MES есть исходный код, так что OTel доработать можно своими силами.
 - Для CRM и Онлайн-магазина прямо не указано есть ли исходный код фронт и бек части, 
но поскольку есть в команде разработчики соответствующих фреймворков, то предполагается, 
что возможно доработать эти системы своими силами так как необходимо.
 - Сложность внедрения и поддержки. 
Настройка трейсинга требует координации между командами, стандартизации инструментария, обучения персонала.
 - расходы на инфраструктуру и инструменты. 
Для работы с трейсингом требуются специализированные инструменты 
(коллекторы, хранилища, системы визуализации), что увеличивает затраты на инфраструктуру.
 - Затраты на хранение и обработку данных. 
Трассировка генерирует большие объёмы данных. Это требует значительных ресурсов для хранения, обработки и анализа. 
Необходимы решения для компрессии, агрегации данных и управления «горячими» и «холодными» данными.

## Аспекты безопасности

 - Разработка и внедрение документа, который определяет правила доступа к системе трейсинга.
 - Настройка идентификации и аутентификации.
 - RBAC модель для управления доступом.
 - Обучение сотрудников.
