# Планирование: анализ, идентификация проблем и поиск решений

## Существующие и потенциальные проблемы

Бизнес-проблемы:

1. Увеличенные сроки получения заказов клиентами (вместо трех недель несколько месяцев)
2. Завал заказов от сторонних продавцов ювелирных изделий
3. Жалобы со стороны сторонних продавцов о неполучении заказов
4. Операторы жалуются на медленную работу MES UI
5. Потеря контрактов из-за проблем с заказами b2c и b2b
6. По ощущениям не хватает (как минимум ошибочных) статусов заказа - ошибка процессинга/производства/расчета и/или других
7. Исследовать возможность распила MES на два сервиса - сервис расчета заказов и сервис производства.


Технические проблемы:

1. Слабая наблюдаемость системы - невозможно отыскать пропавший заказ
2. Отсутствует единая система логирования
3. Отсутствует система трейсинга
4. Отсутствует горизонтальное масштабирование - все сервисы запущены в одном экземпляре
4. CRM API и Shop API хранят данные в одной и той же БД, что затруднит горизонтальное масштабирование
5. Не используется паттерн CQRS для MES, предположение - что он занят расчетами и не может быстро отдать читающую нагрузку
6. Необходима асинхронность в расчетах MES - дополнительно к горизонтальному масштабированию расчитывающих инстансов
7. Нет gateway между клиентским ПО и API для балансировки нагрузки

## Инициативы для устранения нежелательных ситуаций. 

- T-1. Внедрение ELK-стека (ElasticSearch, Logstash, Kibana и Filebeat) для логирования.
- T-2. Внедрение OpenTelemetry и Jaeger для трейсинга
- T-3. Внедрение OpenTelemetry/Prometheus/Grafana для мониторинга.
- T-4. Внедрение кеширования для ускорения страницы заказов MES UI.
- T-5. Реализация асинхронности в расчетах MES
- T-6. Исследовать возможность использования текущего монолита MES в разных ипостасях - расчет и производство
- T-7. Исследование возможности распила MES
- T-8. Развод CRM API и Shop API по своим базам.
- T-9. Паттерн CQRS для MES
- T-10. Внедрение GatewayAPI на границе сервисов

Приоритеты:

- T-2. Внедрение OpenTelemetry и Jaeger для трейсинга
- T-3. Внедрение OpenTelemetry/Prometheus/Grafana для мониторинга.
- T-4. Внедрение кеширования для ускорения страницы заказов MES UI.

Изначально концентрация направлена на прояснение текущей картины.
Логи где-то на серверах есть, просто они не в едином хранилище и доставать их не удобно, но можно. ELK откладываем.
Сначала делаем трейсинг для точных расследований - куда пропадают заказы.
Далее делаем мониторинг для того, чтобы иметь возможность точно понимать состояние системы в количественных измерениях.
Потом, реализуем кеширование как относительно не затратный способ облегчить жизнь операторам.

## Целевая архитектура через полгода

- Через полгода в проекте будет полная наблюдаемость через мониторинг, трейсинг и центральное логирование.
- Будет понятна текущая картина и как она меняется от релиза к релизу в количественных показателях.
- Кеширование MES UI.
- Асинхронные расчеты MES.
- CQRS для MES.
- Отдельные базы CRM API и Shop API.

- Разработана стратегия для горизонтального масштабирования.
