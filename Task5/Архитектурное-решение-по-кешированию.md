# Архитектурное решение по кешированию

## Мотивация

Существуют жалобы от операторов по медленной работе первой страницы MES.
На ней отображается список заказов в работе по статусам — это дашборд с фильтром.
Будем решать эту проблему с помощью кеширования.

В общем случае кеширование улучшает отзывчивость пользовательского интерфейса.
Кеширование уменьшает нагрузку на базу данных или удалённые сервисы.

Разберем моменты, когда кеш не рекомендуется.
В системе MES компании данные меняются достаточно редко, то есть сами параметры заказов почти не меняются с момента оформления,
меняются их статусы. Но это происходит не так быстро и поэтому есть смысл применить кеш.
Конечно, устаревшие данные получать недопустимо, но это решится стратегией инвалидации.
Далее, данные первой страницы запрашиваются операторами часто, они с ней работают постоянно, то есть необходим кеш для чтения.
Совсем секретной информации в заказах нет, если и есть единичные поля, то они будут замаскированы.
Таким образом внедрение кеша для страницы с заказами может принести пользу.


## Предлагаемое решение

### Выбор способа кеширования

Клиентский кеш не подходит, потому что он применяется больше для достаточно статичных данных,
совсем редко изменяемых данных, таких как настройки отображения, шаблоны.

В то время как серверное кеширование позволяет контролировать инвалидацию при изменении статусов,
обеспечивать согласованность данных для всех клиентов, учитывать все параметры дашборда в ключе кеша.
Таким образом здесь больше подходит серверный кеш.

Необходимо использовать паттерн кеширования Cache-Aside.
Нужно привязать ключ кеша к параметрам поиска, которые приходят с клиента.
Во-первых, здесь нужно ускорить операции чтения, так что два паттерна ориентированные на операции записи сразу отпадают (Write-Through, Write-Behind).
Далее, Read-Through подразумевает ту же можель данных, что и в БД. Но мы не знаем заранее, какие запросы на поиск будут приходить с клиента.
То же самое касается патеррна Refresh-ahead - это по структуре такой же Read-Through, но оптимизированный под устранение кеш-промахов,
когда данные асинхронно попадают в кеш из базы заранее, до собственно запросов с клиента.

Таким образом, остается Cache-Aside.

### Ключ кеша
Ключ кеша должен содержать статус (в общем случае и остальные параметры фильтрации, но ограничимся только статусом).
Плюс параметры пагинации. 
Добавляется еще параметр сортировки по времени создания. Косвенно указано, что сортировка есть - "Операторам важно видеть самые новые заказы".

Ключ выглядит так:
```
orders:{status}:{start_position}:{number_on_page}:{sorting_order}
```
Пример, как запрашивают операторы первую страницу готовых к производству заказов:

```text
orders:MANUFACTURING_APPROVED:0:20:desc
```

### Стратегия инвалидации

При изменении статуса необходимо удалить все ключи относящиеся к новому и предыдущему статусу.

В качестве способа инвалидации выбираем комбинированный способ, т.е. инвалидацию на основе изменений дополняем ttl-инвалидацией.
Таким образом базово инвалидация выполняется в запросе на изменение статуса.

ttl-инвалидация нужна как защита в ситуациях, когда упал клиент удаляющий ключи, или, например, для определенного ключа, 
который уже есть в кеше, изменение может никогда не прийти - никто просто может не выполнить изменение порождающее очистку. 
Т.е. кеш должен уметь чистить себя сам, на случай если снаружи его некому чистить, для этого ограничиваем время жизни ключей.

Для надежной очистки кеша при изменении статуса используем механизм Change Data Capture.
База данных ведет лог изменений. Добавляем в схему специальное приложение CDC-агент (Debezium), которое слушает изменения в логе.
Настраиваем CDC-агент на отслеживание изменения статуса и он получает старый и новый статус в своем обработчике,
который и очистит соответствующие ключи.
Использование стороннего компонента для очистки добавит надежности в кеширование. Ключи будут гарантировано удалятся,
при любых манипуляциях с данными из кода. Оно будет асинхронным и не будет влиять на производительность самой базы,
плюс он доставит изменения в кеш в любом случае, если, например, сам кеш упадет.

Формируем два ключа по шаблону:
```
orders:{prev_status}:*
orders:{curr_status}:*
```
Например, при изменении статуса с MANUFACTURING_APPROVED на MANUFACTURING_STARTED

```text
orders:MANUFACTURING_APPROVED:*
orders:MANUFACTURING_STARTED:*
```
